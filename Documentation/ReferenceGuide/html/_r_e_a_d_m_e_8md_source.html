1 Katydid
2 =======
3 
4 Katydid is a data analysis package for Project 8.
5 
6 The various actions Katydid can perform on data are packaged into Processors, which can
7 be setup at runtime into any number of configurations to suit any analysis procedure.

10 Dependencies
11 ------------
12 
13 **External**
14 - CMake (3.1 or better)
15 - Boost (date_time, filesystem, program_options, system, thread; 1.46 or better)
16 - HDF5 (optional, but required by Monarch3 and for outputing to HDF5 files)
17 - Protobuf (optional, but required by Monarch2)
18 - FFTW3 (optional, but required to do FFTs; 3.3 or newer)
19 - ROOT (optional, but required for some processors and outputing to ROOT files; 6.0 or better)
20 - MatIO (optional, but required to read the Matlab files generated by the Tektronix RSA; 1.5.0 or better)

43 Directory Structure
44 -------------------
45 
46 * cmake - Files that CMake uses to build Katydid.
47 * Documentation - Doxygen-based code documentation.
48 * Examples - Configuration files, custom applications, and a tutorial.
49 * External - A few dependencies that are included with the distribution.
50 * Nymph - Submodule
51 * OldAnalysis - Dump for old source files.
52 * Source
53  * Data - Classes that store the data as it gets processed.
54  * Evaluation - Processors for determining how well an analysis is performing.
55  * EventAnalysis - Processors that act on individual events (e.g. tracks, clusters of tracks, etc.).
56  * Executables - Source code for Katydid executables and test programs.
57  * IO - Writer and reader classes.
58  * Simulation - An attempt at integrating a simulation into Katydid (use Locust_MC instead)
59  * SpectrumAnalysis - Processors that act on spectrum-like data.
60  * Time - Processors that act on data in the time domain.
61  * Transform - Processors involved in transforming between the time and frequency domains.
62  * Utility - Generic classes that can be used throughout Katydid.

65 Installing
66 ----------
67 
68 The following steps will build Katydid from scratch. Starting with a terminal window . . .
69 
70 1. Clone the repository and make a build directory as recommended above. You will also have to initialize the submodules.
71  ```
72  $ git clone "https://github.com/project8/Katydid"
73  $ cd Katydid
74  $ git submodule update --init --recursive
75  $ mkdir build
76  ```
77 
78 2. To configure the installation you can use cmake, ccmake, or cmake-gui.
79 
80  For a first configuration, using either ccmake or cmake-gui is highly recommended. The following instructions are for ccmake, but the steps with cmake-gui would be approximately the same.
81  ```
82  $ cd build
83  $ ccmake ..
84  ```
85 
86  You will be prompted to press [c] to configure, and the window will fill up with several options.
87 
88  You should set the CMake variable `CMAKE_BUILD_TYPE` to either `RELEASE`, `STANDARD`, or `DEBUG` (default), in order
89  of how much text output you would like (from least to most) and how much compiler optimization
90  should be performed (from most to least).
91 
92  Variables that start with `Katydid_` will control which parts of Katydid are built.
93 
94  If using Monarch, if you want to read egg3 files, ensure that `Monarch_BUILD_MONARCH3` is set.
95  If you want to read egg2 files, ensure that `Monarch_BUILD_MONARCH2` is set.
96 
97  The install prefix is specified by the CMake variable `CMAKE_INSTALL_PREFIX`.
98  The library, binaries, and header files will be installed in the
99  lib, bin, and include subdirectories. The default install prefix is the
100  build directory.
101 
102  After you've finished, if you've changed anything press [c] again to configure. Then [g] to generate and exit.
103 
104 3. Build and install.
105 
106  ```
107  $ make install
108  ```
109 
110  Or if you want to take advantage of parallel building to get things done faster:
111  ```
112  $ make -j install
113  ```
114 
115  If the compiler runs into errors during the build, first check that you've updated the submodules and that you have all of the required dependencies installed (many are called "optional" on this page, but if you want to build without them you must also specify this in the cmake window). If you made a change to the dependencies or submodules, you may have to wipe the build directory and start again from step 1; simply writing `make install` again will not always work.
